重点发现

1. www 域名返回 Cloudflare 520 高优先级

证据：https://www.hazeseas.help 与 http://www.hazeseas.help 返回 Cloudflare 520，而不是跳转到主域名。主域名 http://hazeseas.help 会 308 到 HTTPS。

影响：如果外部链接、用户输入或搜索引擎尝试访问 www 版本，会遇到服务器错误。它还会削弱域名规范化信号，浪费潜在链接价值。

修复：在 DNS/CDN/Vercel 域名配置中添加 www 版本，并设置永久重定向到 https://hazeseas.help$request_uri。修复后用 curl -I -L https://www.hazeseas.help 验证最终状态为 200，首跳为 301/308。

2. 默认英文前缀已改为 308 永久跳转 已完成

原始证据：https://hazeseas.help/en/weapons 返回 307 Temporary Redirect 到 /weapons；搜索查询中仍能看到历史 /en/weapons 结果。

影响：307 表示临时迁移，搜索引擎可能更久保留 /en/... URL，导致默认语言 URL 信号合并变慢。

修复状态：已在 src/middleware.ts 中加入默认 locale 前缀永久重定向规则。真实请求访问 /en 或 /en/* 时返回 308 到无前缀 URL；内部 rewrite 仍通过保护 header 放行，避免影响默认英文页面渲染。部署后复测 curl -I https://hazeseas.help/en/weapons 应看到 308。

3. fil hreflang 有合规风险 中优先级

证据：页面级 alternates 使用 hreflang="fil"；路由中 locales 为 ["en", "pt-br", "es", "fil"]。

影响：Google 文档要求 hreflang 的语言码使用 ISO 639-1。fil 是常见的 Filipino 语言标签，但不是 ISO 639-1 二字母码；如果 Google 不接受该值，对应菲律宾语 alternate 可能被忽略。

修复：建议验证 Search Console 的 hreflang 报告。如果目标是 Tagalog/Filipino 搜索流量，较稳妥做法是输出 hreflang="tl" 指向当前 /fil/... URL；更彻底的方案是迁移到 /tl/... 并 308 旧 /fil/...。同时保留页面可见语言切换，避免用户体验断裂。

4. sitemap 未携带 hreflang alternates 中优先级

证据：sitemap 有 212 个 <loc>，但没有 xmlns:xhtml，也没有 <xhtml:link>；页面 HTML head 中则完整输出 x-default、en、pt-br、es、fil。

影响：这不是阻断问题，因为 Google 认为 HTML、HTTP Header、sitemap 三种 hreflang 方法等效，当前页面级信号已完整。但 sitemap 不能用于批量校验语言簇，未来如果 head 与 sitemap 分开维护，会产生不一致风险。

修复：二选一：继续采用 HTML head 作为唯一 hreflang 来源，并在代码中注释清楚；或用同一个 getLanguageAlternates 生成 sitemap 的 xhtml:link，确保每个 <url> 都列出自身和所有语言版本。

5. 91 个 meta description 过长 中优先级

证据：212 个页面均有 description，但 91 个超过 160 字符。样例包括 /pt-br/fruits/haze-seas-electricity-fruit 185 字符、/fil/codes/haze-seas-codes 170 字符、/weapons/haze-seas-swords 162 字符。

影响：描述过长容易在搜索结果中截断，尤其是葡语和菲律宾语句子更长，CTR 文案的重点可能被截掉。

修复：优先把过长描述压到 140-155 字符。公式建议：核心查询词 + 页面独特价值 + 更新/奖励/攻略场景，不要把所有子主题都塞进 description。

6. HTML 体积和缓存策略可优化 中优先级

证据：抓取响应时间 p50 424ms、p90 505ms，表现尚可；但首页 HTML 约 248 KB，其他语言首页约 254-255 KB，HTML 响应头为 Cache-Control: private, no-cache, no-store, max-age=0, must-revalidate。

影响：内容站页面大多是静态内容，no-store 会减少 CDN 缓存收益；首页 HTML 体积偏大，会增加首字节后传输和解析压力，对移动端 Core Web Vitals 有潜在影响。

修复：排查 Next.js/next-intl 当前是否触发动态渲染。若内容主要来自本地 MDX，建议使用静态生成或 ISR；首页减少首屏外列表和侧边栏重复数据的 HTML 输出，必要时把非首屏内容延后加载。

7. 本地化回退逻辑有未来重复内容风险 低优先级

证据：src/lib/content.ts 中非默认语言缺内容时会 fallback 到英文；当前抓取没有发现 thin content，也没有发现 hreflang 目标缺失。

影响：如果未来新增英文文章但未同步翻译，sitemap 可能仍生成所有 locale URL，而非英语页面实际显示英文内容，形成跨语言重复和低质量本地化页面。

修复：sitemap 只输出真实存在且已翻译的 locale 页面；未翻译页面返回 404、暂不进入 sitemap，或明确标记为待翻译且不输出 hreflang。避免用英文内容填充非英文 canonical 页面。

8. 结构化数据基础好，但可扩展 FAQ 与更新时间展示 低优先级

证据：抓取到 Organization 212 次、Article 160 次、BreadcrumbList 192 次、ItemList 32 次、WebSite 4 次。

影响：基础结构化数据完整。许多文章有 FAQ 小节，但目前没有 FAQPage schema；文章 schema 有 datePublished/dateModified，页面可见更新时间信号还可以更明确。

修复：对确实有 FAQ 的页面生成 FAQPage schema；在文章标题附近显示“更新日期 / 编辑说明 / 信息来源”，提升信任信号和用户判断效率。
